Why Opponent-Based Features Matter

In MMA, a fighter’s record is context-dependent:
5 wins vs weak opponents → different signal than 5 wins vs top-level opponents
Styles matter: wrestler vs striker, striker vs striker, etc.
Comparing fighters relative to the opponents they faced is powerful:
Average strikes landed vs the same opponent’s previous fights
Strength of schedule, finish rates vs quality of opponents
Basically, it’s like giving the model a “who did they beat?” lens, not just raw stats.
How to Do It Without Overcomplicating
Instead of making a huge opponent table with all possible permutations, you can calculate derived features during feature engineering:
Opponent-Weighted Averages
Example: Fighter A last 3 fights
Average opponent win %
Average opponent finish rate
These are single numbers per fight → easy for ML
Style Matchups
Encode opponent style (wrestler, striker, jiu-jitsu)
Compare to fighter’s style → create features like “style advantage”
Rolling Metrics
Last 3 fights of the opponent, averaged
Combine with fighter’s performance → gives context
You don’t need a separate “opponents table” for the MVP. Just compute these features on the fly from your Fights + FighterStats tables.

Example Feature Pipeline
Feature	How to Compute
avg_opponent_win_pct	Average of last N opponents’ win %
avg_opponent_finish_rate	Average of last N opponents’ finishes
style_advantage	Fighter style vs opponent style encoded as number
reach_diff_vs_opponent	Fighter reach minus opponent reach
age_diff	Fighter age minus opponent age

AI does not handle relationships. Flatten your tables into one ML-ready dataset with all features, including opponent stats and differences.


1️⃣ Why weekly refresh works

UFC fights mostly happen on Saturdays, so stats only change once a week.
No point in refreshing multiple times a day — the data won’t change.
Keeps your CareerStats aggregates (win streaks, KO/Sub counts, last N fights) up-to-date without wasting compute.

2️⃣ Suggested workflow

Saturday night / Sunday morning:
Check for new fights in the last 24 hours (Fights table)
Add them if missing
Update CareerStats:
Recompute rolling aggregates (WinStreak_Last3, WinsByKO_Last5, etc.)
Could be a full recompute or just incremental update
Update ML features:
If you precompute flattened ML table, refresh derived columns (reach_diff, win_pct_diff, KO/Sub differences, style matchups)

3️⃣ Optional: Automate
Use cron job (Linux/macOS) or Windows Task Scheduler
Example cron: run every Sunday at 3 AM

Or in Python with APScheduler if you want the schedule inside your code

NEED TO ADD A FLUKE FLAG FOR MAJOR UPSETS OR BUMS


Ah — that’s a really important concern, and it’s why plain Elo alone can mislead in UFC/MMA. Fighters with a long hiatus could have a high Elo from past wins, but that doesn’t reflect current form. Here’s how you can handle it:

1️⃣ Introduce time decay

Reduce the impact of older fights when calculating Elo.

Example:

Effective Elo Change = EloChange×𝑒−𝜆⋅DaysSinceFight/365 Effective Elo Change=EloChange×e−λ⋅DaysSinceFight/365 λ = decay rate (tunable, e.g., 0.1–0.3)
Fights from years ago contribute less to current Elo

✅ This ensures inactive fighters’ Elo naturally declines over time.

2️⃣ Combine Elo with recent form features

Even with decay, you can add short-term features from CareerStats:
WinStreak_Last3 → last 3 fights win/loss
WinsByKO_Last5 → finishing tendency in recent fights
KO/Sub percentages by round → recent performance patterns
The ML model can then weigh recent form + Elo to avoid overestimating inactive fighters.

3️⃣ Optional: rolling Elo
Instead of using all fights, calculate Elo based on last N fights (e.g., last 10).
This automatically caps influence from very old wins and reflects recent performance better.

4️⃣ Optional: activity flag

Add a LastFightDays feature: number of days since last fight
Use it in ML features or to penalize Elo:
e.g., reduce Elo slightly if fighter hasn’t fought in 2+ years

✅ TL;DR Strategy

Use a decayed or rolling Elo, not plain cumulative Elo.
Include short-term performance features in CareerStats.
Optional activity features to let the model “know” a fighter is rusty.
This way, your Elo remains useful, but doesn’t unfairly favor inactive fighters.

ML Feature Derivation (computed dynamically):
------------------------------------------------
For each fight in Fights:
- Join FighterStats + CareerStats for FighterID
- Join FighterStats + CareerStats for OpponentID
- Compute derived features:
    * ReachDiff = Fighter.Reach - Opponent.Reach
    * EloDiff = Fighter.EloRating - Opponent.EloRating
    * WinStreakDiff = Fighter.WinStreak_Last3 - Opponent.WinStreak_Last3
    * StyleMatchup = Fighter.Style vs Opponent.Style
    * KO_RoundX_Pct_Diff, Sub_RoundX_Pct_Diff, etc.
